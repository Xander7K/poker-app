# 0002 — Authentication strategy: Auth.js + Google OAuth + short-lived JWT for game server

## Status

Accepted (Phase 0). Implementation in Phase 2.

## Context

The application has two components that need to authenticate users:

1. The **Next.js lobby** (`web-lobby`): standard web auth with sessions.
2. The **game server** (`server`): WebSocket connections via Socket.io that
   need to authenticate the user identity at connection time and on reconnect.

Constraints:

- Single developer, side-project pace. Auth code that takes a week to write
  is a week not spent on the poker engine.
- Closed user base (~50 friends). Invitation codes will gate registration.
- No password storage desired (one less attack surface).
- Need a clean way to authenticate WebSocket upgrades, not just HTTP.

## Decision

We use **Auth.js (formerly NextAuth) with the Google OAuth provider** in the
Next.js lobby. After login, when the user joins a table, the lobby's server
side mints a **short-lived JWT** signed with `GAME_JWT_SECRET` containing the
user's id. The game server (Fastify + Socket.io) validates this JWT on
WebSocket connection.

Flow summary:

1. User opens the lobby, clicks "Login with Google".
2. Auth.js handles OAuth callback, creates/looks up user in Postgres, issues
   a session cookie.
3. User navigates to a table; the lobby's server route mints a JWT
   (e.g. 5-minute TTL) with `{ sub: userId, iat, exp }` and returns it to the
   client.
4. The client connects to Socket.io with the JWT in the `auth` payload.
5. Server validates JWT, associates the socket with `userId`.
6. On reconnect inside TTL, the client reuses the JWT. After expiry, the
   client requests a fresh one from the lobby route.

Invitations: a `invitations` table holds single-use codes generated by admin.
The Auth.js `signIn` callback rejects users without a valid invitation.

## Consequences

### Positive

- Auth.js gives us Google OAuth + sessions in well under a day.
- Short-lived JWT for the game server keeps the WebSocket auth logic tiny:
  one `jsonwebtoken.verify` call per connect.
- No password storage, no password reset flow, no email verification flow.
- Auth.js is actively maintained (unlike Lucia, which was deprecated by its
  author in March 2025).

### Negative

- Auth.js is a higher-level abstraction. When we want non-default behavior
  (e.g. attaching custom claims, gating sign-in via invitations), we read its
  callback documentation and configure rather than write the flow ourselves.
- The Google OAuth provider requires a Google Cloud project with consent
  screen + credentials, which is a one-time setup cost in Phase 2.

### Mitigations

- Phase 2 instruction document will include a step-by-step for the Google
  Cloud OAuth setup.
- The JWT minting logic is a thin wrapper we own end to end, so the
  WebSocket side is not at the mercy of Auth.js's internals.

## Alternatives considered

- **Lucia Auth** — rejected. Was deprecated in March 2025 by its author; no
  longer receiving updates. Even before that, Lucia required us to write the
  OAuth callback ourselves (with the `arctic` companion lib), adding ~3 days
  of work for a benefit (low-level control) we do not currently need.
- **Implementing auth from primitives** following the Lucia author's guide
  — rejected for the same time-cost reason. Side-project budget cannot
  absorb this.
- **Better Auth** — viable alternative with TypeScript-first ergonomics.
  Rejected only because Auth.js has wider adoption and more battle-tested
  Google OAuth integration with Next.js. If Auth.js disappoints during
  Phase 2, this is the first fallback.
- **Single Auth.js session for both lobby and game server** (sending the
  session cookie to Fastify) — rejected because it couples the two services
  to a single domain/cookie strategy and complicates running the game server
  on a different host (Fly.io vs Vercel).

## Revisit when

- Auth.js fails to handle a use case we hit in Phase 2 (e.g. claim shape
  doesn't fit, callback API too rigid) — switch to Better Auth.
- We need anonymous "guest" play — reconsider; OAuth-only doesn't fit that.
- We add mobile native (Phase 9) — confirm token flow still applies; likely
  yes, since native apps will hit the same lobby endpoint to exchange tokens.

## Environment variables introduced

- `AUTH_SECRET` — Auth.js session encryption.
- `AUTH_URL` — base URL for callback computation.
- `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` — from Google Cloud Console.
- `GAME_JWT_SECRET` — signing key for short-lived game server JWT. Same as
  `AUTH_SECRET` in dev; can be rotated independently in production.
